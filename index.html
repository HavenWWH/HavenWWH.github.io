removeFromSuperview详解

发表于 2017-09-01

removeFromSuperview

苹果官方文档说明: 从当前视图的父视图和窗口中移除,同时从响应事件的响应者链中移除

此方法做两件事情:

  1. 在树形结构中找到该节点，从树型数据结构中删除该节点及其子节点

  2. 把该对象从响应者链中移除

注意⚠️:

  1. 执行此方法后,会将当前视图视图从它的父视图中移除, 将父视图对它的强引用删除, 如果不存在其他强引用, 则当前视图会自动从内存中释放, 如果还存在其他强引用则不释放
  2. 可以通过设置nil 从内存中移除
  3. 多次执行removeFromSuperview不会报错

iOS响应链之事件传递和响应

发表于 2017-08-18

iOS事件有触摸事件,运动事件,远程事件, 这里特指触摸事件, 也就是平时项目中遇到最多的情况, 每一个触摸事件都是需要有传递和响应两个过程的, 传递事件是为了找出最合适的View, 响应事件是为了找出能响应此事件的View(最合适的View不一定就能响应此事件)

事件传递

事件传递是从父控件传递到子控件的, 当检测到手指触摸（Touch）操作时会将其打包成一个UIEvent对象，并放入当前活动UIApplication的事件队列中, UIApplication会从事件队列中取出触摸事件并传递给UIWindow, 一直传递到一个最合适的视图View来处理事件

传递顺序: UIApplication —>UIWindow —>Subview —> View

系统是怎么找出最合适的View的呢, 依赖hitTest:withEvent:方法去判断点击的点是否在当前View内, 如果是则便利当前View的SubView继续调用hitTest:withEvent:方法, 直到找到最合适的View, 如下图(图片来源网络)

注意⚠️: 遍历subVIews是根据当前subview的index顺序，index越大就越先被访问

事件响应

首先看initial view能否处理这个事件，如果不能则会将事件传递给其上级视图（inital view的superView）；如果上级视图仍然无法处理则会继续往上传递；一直传递到视图控制器view controller，首先判断视图控制器的根视图view是否能处理此事件；如果不能则接着判断该视图控制器能否处理此事件，如果还是不能则继续向上传 递；（对于第二个图视图控制器本身还在另一个视图控制器中，则继续交给父视图控制器的根视图，如果根视图不能处理则交给父视图控制器处理）；一直到 window，如果window还是不能处理此事件则继续交给application处理，如果最后application还是不能处理此事件则将其丢弃

问题: 子View 是 UIView时，如果没有添加手势，点击子 View时，会由其父View来响应，而 子View 是 UIControl 时，子View 没有添加手势，一样不会由 父View 来响应

原因: UIControl默认实现了以下方法

如果某个控件实现了touches…方法，则这个事件将由该控件来接受，如果调用了[supertouches….];就会将事件顺着响应者链条往上传递，传递给上一个响应者；接着就会调用上一个响应者的touches….方法

适配方案:拉伸适配

designW : 设计稿的宽 iphone6为标准, 则设计稿的宽为375, 以6p为标准, 则设计稿的宽为414

designH : 设计稿的高

deviceW : 设备的宽

deviceH : 设备的高

dw : 设计稿中某个控件的位置

vw : 最终在实际的设备中的位置

如果我们以宽适配, 将设计稿的宽拉伸或者缩小到设备的宽, 拉伸或者缩小到比例(x)就是deviceW / designW, 设备宽度除以设计稿宽度

deviceW / designW = vw / dw

vw = dw * deviceW / designW

例如: 设计稿是以6为标准, 那么设计稿的宽designW = 375, 设备宽为deviceW = MIN([UIScreen mainScreen].bounds.size.width, [UIScreen mainScreen].bounds.size.height) 设计稿中控件的宽dw = 100, 那么最终实际设备中的宽vw = [UIScreen mainScreen].bounds.size.width / 375 * 100

如果我们以高适配, 将设计稿的高拉伸或者缩小到设备的宽, 拉伸或者缩小到比例(x)就是deviceH / designH, 设备高度除以设计稿高度

deviceH / designH = vw / dw

vw = dw * deviceH / designH

Hexo搭建

发表于 2017-07-30

Hexo是一款基于Node.js的静态博客框架, 可以在github page平台上托管我们的博客

搭建步骤

第一部分: 环境配置

1. 安装nodejs和nodejs里面的npm工具(因为Hexo是基于nodejs编写的)访问Node.js官网下载

​ node -v 查看nodejs版本

​ npm -v 查看npm版本

2. 安装hexo 安装hexo-deployer插件

sudo npm install -g hexo-cli

​ 安装过程有错误很警告, 暂时不管, 使用hexo -v 查看是否安装成功

npm install hexo-deployer-git —save

第二部分: 建站

1. 初始化Hexo 建立一个网站, Hexo将会在指定文件夹中生成所需要的文件

初始化成功后,文件夹会生成相关文件

_config.yml: 网站的配置信息

node_modules: 依赖包

public：存放生成的页面

scaffolds：生成文章的一些模板, 当您新建文章时，Hexo 会根据 scaffold 来建立文件

source：用来存放你的文章以及资源

themes：主题,Hexo 会根据主题来生成静态页面

2. 打开Hexo服务

​ hero s 或者 hexo server

浏览器打开localhost:4000就可以看到生成的博客

第三部分: 托管到github上

1.github新建一个public仓库

2.生成SSH添加到GitHub

设置SSH密码

3.配置config.yml文件

修改_config.yml文件 配置你的网站

其他命令:

hexo new [layout] 新建一篇文章

hexo new “post title with whitespace” 会在sourse下生成一个md文件, 可以对此文件进行编辑

hexo generate 生成静态文件

hexo publish [layout] 发表草稿

hexo deploy 部署网站

hexo clean 清除缓存文件和已生成的静态文件

其他详细内容参考https://hexo.io/zh-cn/docs/commands

4.提交代码到github

第四部分: 绑定域名

  1. 申请域名, 阿里云 腾讯云。 如果需要申请服务器 需要备案

  2. 解析域名

  3. 在博客目录下天CNAME 。 输入域名, 提交代码到github , 等10分钟, 通过https://www.域名.com访问
