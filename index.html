MQTT详解及使用

发表于 2019-01-30

MQTT

基于发布/订阅模式的轻量级消息传输协议, 底层基于TCP/IP协议

两个端:

Server/Broker: 负责接收消息,过滤消息并转发给消息订阅者

Client: Broker之外的都是客户端，可以发布或订阅消息

三个角色:

发布者Publisher 代理Broker 订阅者Subscriber

发布者和订阅者可以是同一个Client

消息转发的原理:

Publish/Subscribe模式是基于主题(topic)的消息转发系统，每个消息都会被贴上一个特定的topic，

消息并不会直接发送给订阅者(Subscriber)，而是发布者(Publisher) 首先将带有topic的消息发送代理(broker)。

订阅者可以从broker订阅自己相关的topic，当broker接收到发布过来的消息后，

broker开始执行基于topic的过滤，将特定的消息转发给特定的订阅者。

Connect消息结构

基础概念:

MQTT使用过程遇到的一些专业名词解释

CleanSession

client和broker之间是否需要建立持久连接（CleanSession为false表示持久连接）,当使用QoS级别为1或2时, 服务端是否保存客户端连接的session, 包括订阅的主题, Qos信息等, 下次连接的时候就不需要再次订阅主题, 而且还可以再次发送原来未完成的消息

KeepAlive

客户端发起PING Request的时间间隔，确保连接正常。客户端周期性地发送PING请求给broker，broker也会应答此心跳，这种机制可以保证双方知道对方是否还在线

Retained Messages(保留消息)

一台设备意外断开时Broker发送通告给其他相关设备。客户端在建立CONNECT连接时会将遗嘱打包在消息体里。如果这个客户端意外断开连接，broker将会发送遗嘱消息给其他关联的客户端

1个Topic只有唯一的retain保留消息，Broker会保存每个Topic的最后一条retain消息,保留消息可以帮助新的订阅者在订阅主题后立即获取当前状态，而无须等待发布者发送下一条更新,如果不使用保留信息，订阅者在发布者发送间隔期将对当前状态一无所知

最后遗嘱Last will & Testament:

包括Will topic 、will message 、 will Qos、will retain,在客户端 A 进行连接时候，遗嘱消息设定为”offline“，客户端 B 订阅这个遗嘱主题。当 A 异常断开时，客户端 B 会收到这个”offline“的遗嘱消息，从而知道客户端 A 离线了

服务质量(Qos)

最多一次：发送一次，无需回复，消息有可能丢失。（传感器数据）

至少一次：发送者没有收到接收者的反馈，过一段时间会重新发送，消息可能重复,确保消息到达

只有一次：发送过程经过 PUBREC(发布收到)、PUBREL(发布释放)、PUBCOMP(发布完成)，保证一个包只收到一次

CONNACK消息

当broker收到一个CONNECT消息时，broker有义务应答一个CONNACK消息，CONNACK只包含两个数据字段，一个是Session present flag（当前会话标志），另一个是Return code（返回码）

Session Present Flag

broker是否保存和客户端建立过持久会话标识. 服务端是否保存过客户端的session, 如果客户端连接时cleanSession为true, 则当前flag永远为false,

Return Code（返回码）

0 接受连接

1 拒绝连接，不能被接受的协议版本

2 拒绝连接，ID无效

3 拒绝连接，服务器不可用

4 拒绝连接，错误的用户名和密码

5 拒绝连接，未经授权

ios之@property @synthesize @dynamic

发表于 2017-10-19
@property

@property属性 = 带下划线的实例变量 + getter和setter方法

当定义一个属性@property时, 系统会自动生成一个带下划线的实例变量, 还有setter getter方法, 这个过程是在编译的时候进行自动合成的.

@property修饰符有四种

  1. 原子性, 比较消耗性能
  2. 读取权限,
  3. 内存管理, 有assign、strong、weak、copy、unsafe_unretained
  4. set、get方法名, 可以指定setter和getter方法
@synthesize

iOS6之前,@property 与 @synthesize需要搭配使用

上面这个方法表示, 生成一个下划线的属性, 同时自动生成getter和setter方法, 可以用此方法重新定义变量名, 但是不建议这么做

@dynamic 动态合成

@dynamic name;

需要自己定义属性的变量名, setter方法 getter方法

  • setter方法和getter方法实质

  • 访问变量用下划线访问和self访问的区别

    下划线访问是直接访问实例变量的内存位置, 速度相对更快, 但是此时KVO无法触发,因为KVO是基于setter方法, 只有self方式调用才会触发KVO, self使用变量实际上是调用变量getter方法

  • 为什么不能同时重写setter和getter方法?

    如果同时重写了setter和getter方法, 编译会报错, 会提示找不到变量, 因为此时编译器认为开发者想自己管理@property, 但是又没有手动定义变量, 所以报错了

iOS签名认证过程

发表于 2017-10-10
  1. keychain里的“从证书颁发机构请求证书”，Mac上生成一对公私钥, CertificateSigningRequest(公钥L) , Mac上还有一份私钥L存在本地
  2. 苹果后台一个私钥A iOS设备一个公钥A
  3. CertificateSigningRequest(公钥L)上传到开发者账号关联App生成证书, 这个过程苹果后台私钥A与公钥L进行签名, 下载证书到本地
  4. 生成pro描述文件, (APPID, 设备ID, 上一步的证书用私钥A去签名),下载到本地Mac
  5. 编译App时, 用本地私钥L对App进行签名, 和描述文件(provisioning)一起打包安装到手机上
  6. 安装时, 公钥A对描述文件进行验证, 再里面的签名证书(第三步生成的证书)

场景

从AppStore下载App

上传App到AppStore的时候, 苹果后台用私钥对App进行签名, 手机下载后, 手机公钥会验证签名, 通过则可以安装到手机上

removeFromSuperview详解

发表于 2017-09-01

removeFromSuperview

苹果官方文档说明: 从当前视图的父视图和窗口中移除,同时从响应事件的响应者链中移除

此方法做两件事情:

  1. 在树形结构中找到该节点，从树型数据结构中删除该节点及其子节点

  2. 把该对象从响应者链中移除

注意⚠️:

  1. 执行此方法后,会将当前视图视图从它的父视图中移除, 将父视图对它的强引用删除, 如果不存在其他强引用, 则当前视图会自动从内存中释放, 如果还存在其他强引用则不释放
  2. 可以通过设置nil 从内存中移除
  3. 多次执行removeFromSuperview不会报错

iOS响应链之事件传递和响应

发表于 2017-08-18

iOS事件有触摸事件,运动事件,远程事件, 这里特指触摸事件, 也就是平时项目中遇到最多的情况, 每一个触摸事件都是需要有传递和响应两个过程的, 传递事件是为了找出最合适的View, 响应事件是为了找出能响应此事件的View(最合适的View不一定就能响应此事件)

事件传递

事件传递是从父控件传递到子控件的, 当检测到手指触摸（Touch）操作时会将其打包成一个UIEvent对象，并放入当前活动UIApplication的事件队列中, UIApplication会从事件队列中取出触摸事件并传递给UIWindow, 一直传递到一个最合适的视图View来处理事件

传递顺序: UIApplication —>UIWindow —>Subview —> View

系统是怎么找出最合适的View的呢, 依赖hitTest:withEvent:方法去判断点击的点是否在当前View内, 如果是则便利当前View的SubView继续调用hitTest:withEvent:方法, 直到找到最合适的View, 如下图(图片来源网络)

注意⚠️: 遍历subVIews是根据当前subview的index顺序，index越大就越先被访问

事件响应

首先看initial view能否处理这个事件，如果不能则会将事件传递给其上级视图（inital view的superView）；如果上级视图仍然无法处理则会继续往上传递；一直传递到视图控制器view controller，首先判断视图控制器的根视图view是否能处理此事件；如果不能则接着判断该视图控制器能否处理此事件，如果还是不能则继续向上传 递；（对于第二个图视图控制器本身还在另一个视图控制器中，则继续交给父视图控制器的根视图，如果根视图不能处理则交给父视图控制器处理）；一直到 window，如果window还是不能处理此事件则继续交给application处理，如果最后application还是不能处理此事件则将其丢弃

问题: 子View 是 UIView时，如果没有添加手势，点击子 View时，会由其父View来响应，而 子View 是 UIControl 时，子View 没有添加手势，一样不会由 父View 来响应

原因: UIControl默认实现了以下方法

如果某个控件实现了touches…方法，则这个事件将由该控件来接受，如果调用了[supertouches….];就会将事件顺着响应者链条往上传递，传递给上一个响应者；接着就会调用上一个响应者的touches….方法

适配方案:拉伸适配

designW : 设计稿的宽 iphone6为标准, 则设计稿的宽为375, 以6p为标准, 则设计稿的宽为414

designH : 设计稿的高

deviceW : 设备的宽

deviceH : 设备的高

dw : 设计稿中某个控件的位置

vw : 最终在实际的设备中的位置

如果我们以宽适配, 将设计稿的宽拉伸或者缩小到设备的宽, 拉伸或者缩小到比例(x)就是deviceW / designW, 设备宽度除以设计稿宽度

deviceW / designW = vw / dw

vw = dw * deviceW / designW

例如: 设计稿是以6为标准, 那么设计稿的宽designW = 375, 设备宽为deviceW = MIN([UIScreen mainScreen].bounds.size.width, [UIScreen mainScreen].bounds.size.height) 设计稿中控件的宽dw = 100, 那么最终实际设备中的宽vw = [UIScreen mainScreen].bounds.size.width / 375 * 100

如果我们以高适配, 将设计稿的高拉伸或者缩小到设备的宽, 拉伸或者缩小到比例(x)就是deviceH / designH, 设备高度除以设计稿高度

deviceH / designH = vw / dw

vw = dw * deviceH / designH

Hexo是一款基于Node.js的静态博客框架, 可以在github page平台上托管我们的博客

搭建步骤

第一部分: 环境配置

1. 安装nodejs和nodejs里面的npm工具(因为Hexo是基于nodejs编写的)访问Node.js官网下载

​ node -v 查看nodejs版本

​ npm -v 查看npm版本

2. 安装hexo 安装hexo-deployer插件

sudo npm install -g hexo-cli

​ 安装过程有错误很警告, 暂时不管, 使用hexo -v 查看是否安装成功

npm install hexo-deployer-git —save

第二部分: 建站

1. 初始化Hexo 建立一个网站, Hexo将会在指定文件夹中生成所需要的文件

初始化成功后,文件夹会生成相关文件

_config.yml: 网站的配置信息

node_modules: 依赖包

public：存放生成的页面

scaffolds：生成文章的一些模板, 当您新建文章时，Hexo 会根据 scaffold 来建立文件

source：用来存放你的文章以及资源

themes：主题,Hexo 会根据主题来生成静态页面

2. 打开Hexo服务

​ hero s 或者 hexo server

浏览器打开localhost:4000就可以看到生成的博客

第三部分: 托管到github上

1.github新建一个public仓库

2.生成SSH添加到GitHub

设置SSH密码

3.配置config.yml文件

修改_config.yml文件 配置你的网站

其他命令:

hexo new [layout] 新建一篇文章

hexo new “post title with whitespace” 会在sourse下生成一个md文件, 可以对此文件进行编辑

hexo generate 生成静态文件

hexo publish [layout] 发表草稿

hexo deploy 部署网站

hexo clean 清除缓存文件和已生成的静态文件

其他详细内容参考https://hexo.io/zh-cn/docs/commands

4.提交代码到github

第四部分: 绑定域名

  1. 申请域名, 阿里云 腾讯云。 如果需要申请服务器 需要备案

  2. 解析域名

  3. 在博客目录下天CNAME 。 输入域名, 提交代码到github , 等10分钟, 通过https://www.域名.com访问
